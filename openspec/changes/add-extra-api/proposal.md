# Proposal: 新增8个 TuShare 数据接口

## 背景

当前平台已从 TuShare 拉取日线行情、技术因子、申万行业等数据。用户需要扩展以下8类数据以支持更丰富的量化分析维度：


| #   | TuShare API      | 中文名          | 频率                   |
| --- | ---------------- | ------------ | -------------------- |
| 1   | `cyq_perf`       | 每日筹码及胜率      | 日（2018起）             |
| 2   | `cyq_chips`      | 每日筹码分布       | 日（2018起）             |
| 3   | `ccass_hold`     | 中央结算系统持股汇总   | 日（次日9点）              |
| 4   | `hk_hold`        | 沪深港股通持股明细    | 日（2024-08前北向日频，之后季度） |
| 5   | `stk_surv`       | 机构调研记录       | 事件驱动                 |
| 6   | `moneyflow_dc`   | 个股资金流向（东方财富） | 日（2023-09-11起）       |
| 7   | `moneyflow_hsgt` | 沪深港通资金流向     | 日                    |
| 8   | `idx_factor_pro` | 指数技术因子（专业版）  | 日                    |


> ⚠️ `moneyflow_ths`（同花顺个股资金流向）需要 6,000 积分，超出当前账号限制（5,000），已排除。

## 目标

1. **全量历史同步**：从各接口支持的最早日期（一般为2015年或2018年）同步至今
2. **每日增量同步**：写入 `backend/scripts/daily/daily.sh`，支持 `--start-date / --end-date` 参数
3. **后端 API**：每类数据提供 REST 接口，支持 `600118` 格式的 stock code 输入
4. **OpenAPI 更新**：同步更新 openapi.json，方便外部服务（如 AI Agent）调用

## 非目标

- 前端 UI 可视化（本次不包含）
- 实时/Tick 级别数据
- 覆盖 TuShare 积分不足无法访问的字段

## 关键设计决策

### 存储选型

基于数据特征分两类存储：

**Parquet（按 ts_code/year 分区）** — 适合"股票×日期"矩阵型大数据：

- `cyq_perf`：~5000只股 × ~1500交易日，字段少，与 stk_factor_pro 完全同构
- `moneyflow_dc`：同上（历史起点 2023-09-11）
- `idx_factor_pro`：按 index_code/year 分区，70+ 字段

**MongoDB** — 适合小体量或事件型数据：

- `cyq_chips`：按 ts_code/year 分区存为 Parquet，全量历史（~15-35 GB），支持 DuckDB 查询
- `ccass_hold`：仅沪深港通标的，日数量有限
- `hk_hold`：北向持股明细，2024年8月后季度更新
- `stk_surv`：事件型，记录数少，文本内容丰富
- `moneyflow_hsgt`：纯市场级每日汇总，记录数极少

## 成功标准

- 全量历史数据已写入对应存储
- `daily.sh` 新增步骤可在每个交易日稳定运行
- 所有 API 接口通过测试（含 `600118` 格式输入）
- openapi.json 中包含全部新路由
